## 작업 시점 (Perspectives)

모든 작업에서 다음 두 가지 시점을 동시에 적용한다:

- **7년차 Flutter 개발자 시점**: 코드 품질, 아키텍처, 성능, 유지보수성, 안정성을 꼼꼼하게 점검하고 개선한다. 단순히 동작하는 코드가 아니라 견고하고 확장 가능한 코드를 지향한다.
- **10년차 App UI/UX 디자이너 시점**: 사용자 경험, 시각적 완성도, 인터랙션 디테일, 접근성, 최신 디자인 트렌드를 반영한다. 기능 구현뿐 아니라 사용자가 체감하는 품질까지 고려한다.

분석·계획·구현·리뷰 모든 단계에서 두 시점의 관점을 균형 있게 반영한다.

## Coding Convention

### Dart 주석 스타일

- `///` 문서 주석을 사용한다. (`/** */` 사용 금지)
- 종결어미는 명사형으로 통일한다. (`~반환.`, `~삭제.`, `~추가.`)
  - 좋은 예: `/// 저장된 모든 항목을 최신순으로 반환.`
  - 나쁜 예: `/// 저장된 모든 항목을 최신순으로 반환한다.`
- 첫 문장은 마침표로 끝나는 한 줄 요약으로 작성한다.
- 부가 설명이 필요한 경우 빈 `///` 줄 후에 작성한다.
- 파라미터/클래스 참조는 `[paramName]`, `[ClassName]` 대괄호를 사용한다. (`@param` 사용 금지)
- 사용법 예시(코드 블록)는 주석에 포함하지 않는다.
- 이름만으로 의미가 충분한 경우에도 최소한 한 줄 요약은 작성한다.
- 클래스 주석에는 역할과 관련 클래스 간의 관계를 명시한다.

## 기능 구현 시 영향 범위 점검

기능을 추가·수정·삭제할 때, 해당 기능이 앱 전체에 미치는 **연쇄 영향(side effects)** 을 반드시 사전에 분석하고 처리한다.

### 데이터 변경 시 참조 정합성

- 데이터를 생성·수정·삭제할 때, 해당 데이터를 **참조하는 모든 곳**을 추적한다.
  - DB 레코드, 메모리 리스트, 파일 시스템, 캐시 등 저장소 계층별로 확인한다.
  - 다른 모델이 ID·경로 등으로 참조하는 경우 고아 참조(orphan reference)가 발생하지 않는지 점검한다.
- 좋은 예: 다운로드 기록 삭제 시 → Hive 레코드 삭제 + 실제 파일 삭제 + 플레이리스트 참조 정리 + 좋아요 목록 반영 여부 확인.

### 현재 활성 상태와의 충돌

- 삭제·수정 대상이 **현재 사용 중**(재생 중, 편집 중, 네트워크 요청 중 등)인지 확인한다.
  - 사용 중이라면 먼저 해당 상태를 안전하게 해제(정지, 취소 등)한 뒤 작업을 수행한다.
- 좋은 예: 재생 중인 곡을 삭제할 때 → 재생 정지(또는 큐에서 제거) 후 파일 삭제.

### 동일·유사 데이터 중복 존재

- 같은 논리적 데이터가 **물리적으로 여러 개** 존재할 수 있는 시나리오를 고려한다.
  - 하나에 대한 조작이 다른 것에 의도치 않게 영향을 주거나, 반대로 영향을 주지 못하는 경우를 점검한다.
- 좋은 예: 같은 영상을 2회 다운로드 → 파일명이 다르므로 하나 삭제 시 나머지는 유지되는지 확인.

### 사용자 확인 및 되돌릴 수 없는 작업

- **비가역적 작업**(파일 삭제, DB 레코드 영구 삭제 등)은 반드시 사용자에게 결과를 명시하고 확인을 받는다.
- 확인 다이얼로그에는 **무엇이 삭제되는지** 구체적으로 표시한다.

## Flutter UI 제스처 설계 원칙

- `GestureDetector`로 넓은 영역을 감쌀 때, 내부에 `IconButton`·`InkWell` 등 자체 탭 핸들러를 가진 위젯이 있으면 **제스처 경쟁(gesture arena)** 이 발생하여 양쪽 모두 동작하지 않을 수 있다.
- **탭 영역을 분리**한다: 네비게이션용 탭 영역과 액션 버튼 영역을 같은 `GestureDetector`로 감싸지 않는다.
  - 나쁜 예: `GestureDetector(onTap: navigate, child: Row([thumbnail, text, IconButton, IconButton]))`
  - 좋은 예: `Row([Expanded(GestureDetector(onTap: navigate, child: Row([thumbnail, text]))), IconButton, IconButton])`
- 빈 영역도 탭을 감지해야 하면 `behavior: HitTestBehavior.opaque`를 명시한다.
- 코드 작성 시 **먼저 위젯 트리 내 제스처 핸들러 중첩 여부를 확인**한 뒤 구현한다.

## Git 관리

- 새로 생성한 파일은 `git add`를 실행하여 추적 대상에 포함시킨다.
- 기존 파일 수정 시에도 변경 사항을 스테이징한다.
- 커밋 메시지는 **한 줄 제목만** 작성한다. 본문(부연 설명, 불릿 목록)은 포함하지 않는다.
- **Conventional Commits** 형식을 사용한다: `<type>: <한국어 설명>`
  - `feat`: 새 기능 추가
  - `fix`: 버그 수정
  - `perf`: 성능 개선
  - `refactor`: 동작 변경 없는 코드 구조 개선
  - `style`: 포매팅, 코드 스타일 정리
  - `docs`: 문서 변경
  - `test`: 테스트 추가/수정
  - `chore`: 빌드, 설정, 의존성 등 기타
  - 좋은 예: `fix: AudioPlayer·HttpClient 리소스 누수 수정`
  - 나쁜 예: 제목 아래에 `- 변경 내용 1\n- 변경 내용 2` 등 본문 추가
